# VD Outline Workflow: System 2 (Compressed)
## Version: 1.0.0-compressed | Source: vd-outline-workflow-system-2-v1.0.0.md

---

## TRIGGER CONDITIONS

Use System 2 only when user provides ALL of:
- Four required inputs (page type, assigned pillar, primary topic/entity concept, target URL slug)
- Research material (any form: brief, notes, transcript, school profile, source packet, etc.)

Missing research material → use System 1.
Missing any required input → stop and request it.

---

## FOUR REQUIRED INPUTS

| Input | Use |
|---|---|
| Page type | Structural pattern, depth, required sections |
| Assigned pillar | Pillar Architecture scope, keyword territory, cluster role |
| Primary topic / entity concept | Authority concept for the page |
| Target URL slug | Entity signal; informs title, H1, internal linking |

**NEVER infer missing required inputs.**

---

## CORE CONTROL HIERARCHY

1. User's explicit task instructions
2. This System 2 workflow
3. Page Creation Architecture (page architecture + creation standards)
4. Pillar Architecture (pillar scope, keyword territory, cluster strategy)
5. Research material (factual boundaries for this page)
6. Brand positioning → Language control → AI extractability → Tone/style → Keyword type map → Keyword audit → Fact-checking

**Research material controls facts. Pillar Architecture controls strategy. Pillar Architecture cannot override research material's factual boundaries.**

---

## HARD RULES (ZERO-COMPRESSION ZONE)

- NEVER add facts, stats, examples, school/program claims, VA policy claims, benefit claims, salary/career data not in the research material.
- NEVER search the web unless user explicitly authorizes it.
- NEVER produce a formal Fact-Check Report or Markdown Fact-Check Table.
- NEVER produce a draft-stage Keyword Audit Report.
- NEVER run Copy QA unless user explicitly requests it.
- NEVER invent writer names, publish dates, inbound links, or source details.
- NEVER force unsupported Pillar Architecture angles into the outline.
- If any EAS dimension projects below 4.0 → revise before writer handoff.
- Outline Keyword Coverage Score below 3.5 → revise before writer handoff.
- Use "Not provided / Not applicable / Needs confirmation / Research gap / Source gap" when appropriate.

---

## RESEARCH MATERIAL USE

Review material to identify:
- Supported: facts, examples, stats, school/program details, VA/benefit/policy refs, career/salary data, quotable points
- Unsupported: unclear, outdated, or risky claims
- Gaps: research gaps, source gaps

Outline sections may only be built around supported items. Unsupported Pillar Architecture angles → flag as gap, not a required section.

---

## PREFLIGHT STOP CONDITIONS

Stop if:
- Any required input is missing
- Research material is unreadable/unparseable
- Assigned pillar not in Pillar Architecture
- Research material does not support the requested topic
- Page type conflicts with topic in a way that makes the outline unreliable
- User requests central factual claims the research does not support

State exactly what is missing. Do not proceed by guessing. Non-blocking gaps → continue and flag.

## TITLE TAG RULES

- 3 options required
- Each: under 60 characters; brand as suffix (not prefix); scored 1–5 per Page Creation Architecture rubric; pillar/topic confirmed
- At least 1 must score 4 or 5 before proceeding
- If none score 4–5, revise before continuing

---

## H1 + META DESCRIPTION RULES

H1 (2 options each):
- Same topic as selected title tag; not verbatim repeat; veteran-reader first, keyword second; platform identity preserved

Meta description (1 draft):
- Primary keyword/concept naturally included; platform identity signal present; no benefit guarantees; under 155 characters; use "housing allowance" unless search context requires otherwise

---

## OPENING 150 WORDS BLUEPRINT FORMAT

```
OPENING 150 WORDS BLUEPRINT

Writer must establish within first 150 words:
[ ] 1. What this page is about
[ ] 2. Who this page is for
[ ] 3. What the reader can do with this information
[ ] 4. VeteranDegrees.com's role as a guidance and verification platform
[ ] 5. Hybrid program connection [when relevant + research-supported]

Primary keyword/concept within first 100 words: [term]
Required caution: [benefit, policy, school, or source limitation]
Research boundary: [what research supports for the opening]
```

This is a writing spec. Do not write the opening paragraph.

---

## SECTION BLOCK FORMAT

```
SECTION [N]: [Suggested H2 Text]

Word Count Target: [range]
Pillar Function: [EAS/entity function]
Keyword Signal: [secondary keyword or semantic term]
AI Extractability: [Yes / No — if yes, specify self-contained answer required]
Internal Link: [target + anchor text, or None required]
Hybrid Anchor Required: [Yes / No — if yes, specify]
Research Support: [what research supports for this section]
Research Gap / Source Gap: [unsupported item, if any]

WHAT TO WRITE:
[Specific writing direction. Min 3 sentences. Cover, avoid, accomplish, source boundary.]

WHY THIS SECTION EXISTS:
[1–2 sentences on entity/EAS/reader/search-intent function.]

ANTI-PATTERNS TO AVOID:
[Section-specific risks.]
```

Do not write final body copy. Hybrid anchor only where relevant to entity alignment, not in every section.

---

## FAQ BLUEPRINT FORMAT

```
FAQ SECTION BLUEPRINT

[ ] Q: [search-style question]
    Maps to: [long-tail keyword or Not available]
    Answer must include: [one-sentence spec]
    Research boundary: [what research supports]
    AI anchor: [Yes / No]
```

Do not write full answers. If FAQ is not appropriate, state why.

## ANTI-PATTERN WARNING SCOPE

Warnings must be page-specific. Cover risks including:
- Platform identity drift
- Benefit guarantee language
- Overuse of "BAH"
- Forced hybrid framing
- Unsupported Pillar Architecture seed claims
- Unsupported research extrapolation
- Generic veteran education content
- Keyword stuffing
- Vague evidence summary language
- Invented examples or statistics
- Provider-style language
